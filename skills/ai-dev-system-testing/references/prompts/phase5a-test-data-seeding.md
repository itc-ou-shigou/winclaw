You are executing Phase 5A — a system-level test data seeding phase.
Your goal is to analyze the project's model/entity classes and create test data
so that Phase 5B (API testing) and Phase 5C (UI testing) can use real entity IDs.

**No hardcoded IDs, no hardcoded schemas. Everything is discovered dynamically.**

## Step 2: Model Class Analysis

Discover and analyze ALL model/entity classes in the backend directory.

### 2a. Find Model Files

Use Glob to search for model files. Try these patterns (adapt to the backend directory):

- `**/models.py`, `**/models/*.py` (Django)
- `**/model.py`, `**/models/**/*.py` (Flask/FastAPI)
- `**/schema.prisma` (Prisma)
- `**/*.entity.ts` (TypeORM/NestJS)
- `**/entities/**/*.java`, `**/model/**/*.java` (Spring Boot)
- `**/models/**/*.rb` (Rails)

### 2b. Parse Each Model

For each model/entity file found, extract:

| Field | Description |
|-------|-------------|
| **Entity name** | Class/table name |
| **Fields** | Name, type, nullable, default value |
| **Primary key** | Field name and type (auto-increment, UUID, etc.) |
| **Foreign keys** | Field name, referenced entity, referenced field |
| **Required fields** | Non-nullable fields without defaults |
| **Unique constraints** | Fields with unique=True or unique indexes |

#### ORM-Specific Patterns:

**SQLAlchemy:**
```python
Column(Integer, primary_key=True)
Column(String, nullable=False)
Column(Integer, ForeignKey('users.id'))
relationship("User", back_populates="posts")
```

**Django ORM:**
```python
models.CharField(max_length=200)
models.ForeignKey(User, on_delete=models.CASCADE)
models.IntegerField(default=0)
```

**Prisma:**
```prisma
model User {
  id    Int    @id @default(autoincrement())
  email String @unique
  posts Post[]
}
```

**TypeORM:**
```typescript
@PrimaryGeneratedColumn()
@Column({ nullable: false })
@ManyToOne(() => User, user => user.posts)
```

Create a mental catalog of all entities with their fields and relationships.

## Step 4: Dependency Graph & Data Creation

### 4a. Build Dependency Graph

From the FK relationships discovered in Step 2, build a directed acyclic graph:
- Node = entity
- Edge = FK dependency (entity A depends on entity B if A has FK to B)

Compute a **topological sort** to determine creation order.
Entities with no FK dependencies come first.

### 4b. Create Test Data (Priority-Tiered)

Data creation follows a priority system. **Always complete P0 first**, then proceed through P1-P5 if BUSINESS_LOGIC_TESTCASES.md was loaded in Step 1.5.

**Before starting each priority tier, check elapsed time.** If remaining time < 300 seconds, skip remaining tiers and proceed directly to Step 5 (save results).

#### P0: Basic CRUD (MANDATORY — current behavior)

For each entity in topological order, create at least 1 test record.

**Priority of creation methods:**

1. **POST API** (preferred if backend is running):
   - Use `curl` via Bash tool to call REST API endpoints
   - Discover endpoints from Swagger/OpenAPI spec if available
   - Try common patterns: `POST /api/v1/{entity}`, `POST /api/{entity}s`
   - Include proper authentication headers if required

2. **Direct DB INSERT** (fallback):
   - Generate a Python script using the project's ORM
   - Or use raw SQL via `python3` + database driver
   - Execute via Bash tool

**Authentication/User creation is ALWAYS first:**
- Register a test user: `POST /api/v1/auth/register` or similar
- If registration endpoint doesn't exist, create user via DB INSERT
- Login to get JWT/session token: `POST /api/v1/auth/login` or similar
- Use the token for all subsequent API calls

#### P2: Validation Scenarios (if BUSINESS_LOGIC_TESTCASES.md has Validation Rules)

For each validation rule from BUSINESS_LOGIC_TESTCASES.md:
- **Do NOT create database records** — validation tests use invalid data
- Build payload objects with invalid values from the "Invalid Example" column
- Record them in `scenarios.validation` array:
```json

  "payload": {"email": "not-email", "password": "Test@123456"},
  "expected_status": 422 }
```

#### P3: State Machine Data (if BUSINESS_LOGIC_TESTCASES.md has State Machines)

For each state machine entity, create records in different states:
- Create one record per interesting state (e.g., order in "pending", another in "confirmed")
- Use API or direct DB to set the state field
- Record transition scenarios in `scenarios.state_transitions`:
```json
{ "entity": "orders", "record_id": 10, "from": "pending", "to": "confirmed",
  "endpoint": "PATCH /api/v1/orders/10/status", "expected_status": 200 }
```
Include both VALID and INVALID transitions.

#### P4: Business Constraint Data (if BUSINESS_LOGIC_TESTCASES.md has Business Constraints)

Create setup data for constraint tests:
- e.g., product with `stock=0` for "insufficient stock" test
- e.g., user with duplicate email for "already exists" test
- Record in `scenarios.business_constraints`:
```json
{ "constraint": "stock_check", "endpoint": "POST /api/v1/orders",
  "setup": "product with stock=0", "expected_status": 400 }
```

#### P5: Error Path Data (if remaining time allows)

Create edge case data for error handling tests.
- Non-existent IDs for 404 tests
- Malformed inputs for 400 tests

### 4c. Record All Created Data

For every entity created, record:
- Entity name
- ID (from API response or DB)
- Key fields (email, name, etc.)
- Creation method (`POST /api/...` or `DB INSERT`)
- Any associated IDs (e.g., user_id for child entities)

## Critical Rules

1. **No hardcoded schemas** — discover everything from the codebase
2. **No browser automation** — use only Bash (curl/python3), Read, Glob, Grep
3. **Topological order** — always respect FK dependencies
4. **Real IDs only** — record actual IDs returned by API/DB, never guess
5. **Idempotent** — if data already exists (e.g., user with same email), handle gracefully
6. **Must produce `test-logs/test_data_seed.json`** — this is the mandatory output
7. **Must produce `test-logs/phase5b_test_user.json`** — Phase 5B/5C depend on this
8. **Use `encoding='utf-8'`** in all Python file operations (Windows compatibility)
9. **P0 (Basic CRUD) is always mandatory** — complete it before any scenario data
10. **Time budget management** — check remaining time before each priority tier, skip if < 300s remaining
11. **Graceful degradation** — if BUSINESS_LOGIC_TESTCASES.md is missing or incomplete, fall back to P0-only behavior
12. **Scenario data format** — every `scenarios` entry must include `expected_status` for Phase 5B/5C validation
